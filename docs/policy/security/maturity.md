# Maturity Platform Levels

As technology continues to advance at a rapid pace, it is essential to have a framework for evaluating the technical maturity of the platform.
Technical Maturity Levels provide a standardized way to assess the current state of the work made on any [Rome program](../rome-programs/introduction.md) and identify areas that need improvement to achieve optimal performance.

Technical Maturity Levels allows Rome programs to meet to a defined set of quality standards by providing clear guidelines and expectation both internally and externally.​
This allows Rome builders (us) to move fast and reduce cognitive load, it also enables the platform and teams to be prepared to adapt faster and more reliably to (planned or unplanned) changes.​

Technical Maturity Levels are categorized into several stages and categories, each representing a level of maturity.

The use of Technical Maturity Levels enables Rome contributors in partnership with the Rome Leadership, to evaluate programs based on their specific needs and make informed decisions about which area to focus on.
This framework can also help developers and engineers understand the current state of their product's development and determine what needs to be done to achieve optimal performance and functionality.

In this context, it is important to note that technical maturity levels are not static and can change over time as the platform evolves. Therefore, it is necessary to periodically evaluate and update the platform's maturity level to ensure it remains relevant and optimized for its intended use.

## Maturity categories

Maturity criteria are grouped together under various categories to better represent each technical goal.

- **Engineering Practices**: This category contains criteria defining standards of quality required to produce the best products, services and tools.
These standards contain a set of policies, practices, and tools that are necessary to make the software system useful for as long as it needs to be used and allowing collaboration across teams.
- **Live Operations**: As Rome is providing a set of managed products and services to our users, we need to ensure operational standards. These criteria regroup activities pertaining to Reliability, Performance and Stability
- **Platform Compliance**: This category regroups all criteria required to provide a coherent and cohesive experience for our partners and users.
- **Security**: As Technology evolves, we need to ensure our products are built with security in mind.

## Level definitions

The Technical Maturity Levels are defined based on the product lifecycle, from inception to general availability, as defined in the table below.

In order to obtain a level badge, all criteria of this level need to be completed, as well as all prior levels' criteria.

| Level | Name                                   | Description                                        |
|-------|--------------------------------------------------|---------------------------------------------------------|
| 1 | Initial design                            | Required design materials is available and approved.     |
| 2 | Technical foundations                       | Product is ready to start integrating with the platform. |
| 3 | Platform integration | Product is ready for early adopters' feedback.          |
| 4 | Soft launch | Service can be opened to users under Beta conditions. |
| 5 | Launch   | Service stability improved from feedback. |
| 6 | Generally available | Service is live, scalable, and resilient. |
